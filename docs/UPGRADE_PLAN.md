# HomeMind 升级计划书

更新日期：2026-04-23

## 当前完成情况（2026-04-23）

为避免本计划书只描述目标、不反映现状，先同步当前代码库真实进度：

### 已完成

- Web 端家庭场景主链路已可运行，当前功能可以完整接入 mock 数据。
- 设备控制、场景模式、规则页、最小 TAP 规则 CRUD 已补齐到现有界面中。
- 已迁入并接通 SVG 户型图上传、元数据存储、设备映射导入、删除、编辑的基础能力。
- 已补充 SSE、Schema 基础校验、上下文压缩相关接口与脚本层支持。
- 启动脚本、编码配置、文档目录已完成同步整理。

### 部分完成

- 新版 UI 原型已适配到现有页面，但对话区压缩、无效留白治理、logo 与户型图渲染仍在修正。
- 端云协同只完成了方向性设计和局部接口，未形成完整生产链路。
- 本地记忆、偏好写回、规则执行回写已有骨架，但尚未完全贯通到用户可见体验。

### 未完成

- 端侧资源约束的量化验证、benchmark、并发与时延证明尚未完成。
- 方言识别、口语归一化、结构化长期记忆、规则冲突检测与统计仍未落地。
- 赛题要求中的跨场景扩展能力目前仍缺代码级证明。

## 1. 计划目标

本计划用于把 HomeMind 从“可运行的原型系统”升级为“可持续扩展的智能家居中央指令器”，重点不是堆功能，而是解决以下四类问题：

- 当前主链路可跑通，但端云协同、上下文持久化、规则自动化、空间配置能力还不完整。
- 端侧资源必须受控，业务算法模型内存占用目标控制在 `1G-2G`，上限不超过 `4G`，LLM 可放云端。
- 前端已有交互壳，但缺少规则配置、记忆写回、空间设备配置等产品化能力。
- 需要吸收 `D:\HausRAGen-main` 中已经验证过的 SVG 上传、设备映射、SSE、Token 压缩、Schema 校验等能力，但不能生搬硬套。

## 2. 当前基线

结合当前 HomeMind 代码库，已具备的基础能力包括：

- 基于 `BSR -> LSR -> LLM` 的自然语言理解主链路。
- 基于 `DeviceController`、`SceneSwitcher` 的统一执行层。
- 本地知识库、偏好记录、基础记忆雏形。
- `Flask + Socket.IO` 的 Web 交互面板。
- DQN 轻量主动推荐骨架。
- mock 数据链路已修复，当前功能可以完整应用 mock 数据运行。

当前缺口包括：

- 端云协同只有方向，没有成型的稳定路由和隐私边界。
- 规则自动化仍偏弱，尚未形成最小 TAP 引擎。
- 空间配置能力缺失，无法把“户型图 + 设备位置 + 规则触发”连成一体。
- 上下文持久化结构还不够清晰，反馈写回不够结构化。
- 对方言、口语化、模糊意图的处理还不够贴近真实家庭环境。

## 3. 升级原则

### 3.1 不推翻主链路，只做增量升级

- 保留当前自然语言理解主链路。
- 保留当前执行层和 mock 运行方式。
- TAP 只作为自动化补充层，不替代全部 AI 决策。
- 前端优先做适配和增强，不做大规模重构。

### 3.2 云端只负责理解建议，执行权留在端侧

- 云端 LLM 只输出固定 JSON 建议，不直接控制设备。
- 端侧必须完成设备白名单、参数范围、动作合法性校验。
- 高风险动作必须支持确认或降级。

### 3.3 资源约束优先

- 端侧不部署大模型。
- 端侧承担召回、排序、压缩、脱敏、规则执行、设备控制。
- 云端承担模糊意图理解、补全和澄清。
- 业务算法内存目标控制在 `1G-2G`，预留系统和 UI 资源空间。

### 3.4 用户价值优先，不强行做“解释性展示”

你已经明确指出“让用户看见系统如何理解”不是必要目标，因此 P1 阶段不以“完整展示推理过程”为主，而改为：

- 用户可修正结果。
- 用户可管理规则。
- 用户偏好与上下文可持久化。
- 用户能感知系统记住了什么、还能改什么。

## 4. 总体架构

### 4.1 升级后数据流

```text
用户输入 / 语音输入
-> 端侧标准化（口语/方言归一化）
-> 本地规则直达判断
-> BSR 召回
-> LSR 精排
-> Router 判断是否上云
-> 端侧最小必要上下文压缩与脱敏
-> 云端 LLM 输出固定 JSON
-> 端侧 Schema 校验与白名单校验
-> DeviceController / SceneSwitcher 执行
-> 反馈、偏好、上下文结构化写回
-> 若命中 TAP 规则则由规则引擎持续自动执行
```

### 4.2 模块分层

1. 交互层
- Web UI
- 语音输入
- 反馈入口
- 规则管理页
- 户型图与设备映射页

2. 理解层
- 方言/口语归一化
- BSR
- LSR
- Inference Router
- Cloud LLM Decider

3. 安全与成本层
- 隐私脱敏器
- 上下文压缩器
- Schema 校验器
- 设备白名单与参数校验

4. 记忆层
- 会话上下文
- 结构化偏好
- 本地长期记忆

5. 自动化层
- 最小 TAP 引擎
- 规则注册表
- 冲突检测
- DQN 推荐

6. 执行层
- DeviceController
- SceneSwitcher
- 协议网关

## 5. 关键技术选型与结论

本节按 AGENTS 要求记录主要技术决策，避免只写结论不写取舍。

### 5.1 前端升级方式

方案 A：继续沿用当前 `web/client/index.html` 单页增强  
优点：
- 改动范围小，能保留现有业务逻辑和 Socket.IO 通信。
- 适合先把 UI 原型、规则页、户型图页快速接入。
- 维护成本低，适合当前项目阶段。  
缺点：
- 随着功能增多，可维护性会下降。
- 状态和视图逻辑容易继续堆在一个文件里。

方案 B：拆到 React/Vue 组件化前端  
优点：
- 结构更清晰，后续规则页、地图页、设置页更好维护。
- 更适合复杂状态和复用组件。  
缺点：
- 会改动构建链路和前端结构。
- 当前用户要求是适配现有代码库，不重写业务逻辑，迁移成本偏高。

最终选择：短中期采用方案 A。  
理由：当前阶段目标是先补齐产品能力，不是重构技术栈。后续若页面继续扩张，再考虑把 UI 从单文件中拆出。

### 5.2 对话流式传输方式

方案 A：继续使用现有 `Socket.IO / WebSocket`  
优点：
- HomeMind 当前已接入，链路真实存在。
- 双向通信能力更强，适合聊天、状态回传、日志推送。  
缺点：
- 前端和服务端都要保持连接状态管理。

方案 B：新增 SSE  
优点：
- 对单向流式文本输出更轻量。
- 类似 `HausRAGen-main` 的 `/api/generate-tap` 很适合用 SSE。  
缺点：
- 当前 HomeMind 已有 WebSocket，再加 SSE 会让传输层重复。
- 不适合需要双向实时控制的场景。

最终选择：对话交互主通道继续用 `Socket.IO`，不把 SSE 作为主方案。  
补充：SSE 可以作为“长文本生成 / TAP 生成 / 导出任务”的补充接口保留，但不建议在当前阶段替换主对话通道。

### 5.3 Schema 校验方式

方案 A：继续使用手写字典校验和正则校验  
优点：
- 实现快。
- 没有新增依赖。  
缺点：
- 规则分散，边界容易漏。
- 难以统一复用在命令、规则、设备映射三个方向。

方案 B：结构化 Schema 校验  
可行实现：
- 命令 JSON：`Pydantic` 或等价数据模型
- TAP YAML：`PyYAML + 自定义规则校验`
- 设备映射：统一 tuple/object 双格式归一化校验  
优点：
- 能把“解析、校验、报错”统一起来。
- 更适合端云协同固定 JSON 输出。  
缺点：
- 初期建模成本更高。

最终选择：采用方案 B。  
理由：端云协同的安全性依赖结构化校验，不能继续靠散落的 if/else。

### 5.4 户型图与设备空间建模方式

方案 A：SVG 户型图 + 设备映射 JSON  
优点：
- 可解析尺寸，可做区域和设备可视化。
- 与 `HausRAGen-main` 已验证方案兼容。
- 便于后续叠加规则、空间感知、房间级推荐。  
缺点：
- 需要维护 SVG 上传、元数据、设备坐标计算。

方案 B：PNG/JPG 平面图 + 手工坐标点位  
优点：
- 实现更简单。  
缺点：
- 尺寸和区域语义弱，后续自动定位与编辑体验差。

最终选择：采用方案 A。  
理由：空间配置一旦要进入产品层，SVG 是更稳妥的基础能力。

### 5.5 Token 压缩方式

方案 A：规则式压缩  
手段：
- 候选剪枝
- 上下文去重
- 摘要压缩
- 输出固定 JSON  
优点：
- 对端侧资源友好。
- 可解释、稳定、无需端侧再加模型。  
缺点：
- 泛化性不如学习型压缩。

方案 B：端侧小模型做上下文摘要  
优点：
- 对复杂上下文更灵活。  
缺点：
- 增加内存和推理成本，不利于 `1G-2G` 目标。

最终选择：采用方案 A。  
理由：当前资源约束非常明确，优先做可控的规则式压缩。

### 5.6 TAP 演进方式

方案 A：全面改造成 TAP-first 系统  
优点：
- 自动化逻辑统一。  
缺点：
- 会破坏现有自然语言主链路。
- 模糊意图场景反而更难处理。

方案 B：最小 TAP 引擎作为并行补充层  
优点：
- 不影响现有交互主链路。
- 适合定时、状态、环境触发等明确自动化场景。  
缺点：
- 需要处理 TAP 与 AI 决策的优先级关系。

最终选择：采用方案 B。  
理由：这是最符合 HomeMind 当前状态的渐进路线。

## 6. 升级项设计

## 6.1 前端 UI 与交互优化

目标：

- 让对话、规则、设备、空间配置进入同一产品界面。
- 不强调展示完整推理过程，强调“可修正、可配置、可管理”。

建设内容：

1. 聊天页优化
- 继续适配现有原型布局。
- 保留进度条、右侧面板、日志面板、输入区。
- 右侧面板增加规则入口、设备入口、记忆入口。

2. 反馈与纠正入口
- 对执行结果提供“接受 / 更正 / 拒绝 / 下次别这样”的快速反馈。
- 更正结果写入本地偏好和长期记忆。

3. 规则管理页
- 查看规则列表。
- 启用 / 禁用 / 删除规则。
- 新建最小规则。
- 查看规则命中次数、最近执行时间、冲突状态。

4. 户型图与设备配置页
- 上传 SVG 户型图。
- 可视化查看设备点位。
- 导入 / 编辑设备映射 JSON。
- 支持设备与房间的绑定关系展示。

5. 隐私提示与云调用提示
- 显示是否调用云端。
- 显示发送的是“摘要上下文”而非原始家庭数据。

预期价值：

- 提升可操作性，而不是堆解释性。
- 提升规则配置与空间配置能力。
- 让用户真正能管理系统，而不是只能聊天。

## 6.2 端云协同 AI 部署

目标：

- 明确命令本地直达。
- 模糊意图由云端协同。
- 云端只输出固定结构建议。
- 内存和 Token 成本可控。

设计原则：

1. Router 分流
- `LSR Top1 >= 0.85`：本地直接执行。
- `0.55 <= score < 0.85`：上云裁决。
- `< 0.55`：优先澄清。
- 断网：强制 fallback。

2. 最小必要上下文
- 端侧只上传时间段、环境摘要、占用人数、当前场景、Top-K 候选、少量偏好摘要。
- 不上传完整设备日志、完整家庭成员细节、完整历史对话。

3. 固定 JSON 输出

```json
{
  "action": "device_control",
  "target": "air_conditioner",
  "operation": "on",
  "params": {
    "temperature": 26
  },
  "confidence": 0.91
}
```

4. 端侧二次校验
- Schema 校验。
- 设备白名单校验。
- 参数范围校验。
- 高风险动作确认。

### 算法优化方向

在“不超过 1G-2G 业务算法内存、LLM 可云端”的前提下，建议算法层做以下优化：

1. 召回层轻量化
- BSR 保持关键词 / 模板 / 规则召回。
- 避免引入大规模端侧向量索引。
- 若引入向量能力，只保留小规模偏好记忆检索。

2. 排序层轻量化
- LSR 保持浅层特征模型或小 MLP。
- 避免端侧部署大 reranker。

3. 路由优先
- 通过 Router 限制上云频率。
- 明确命令不走云。

4. 上下文压缩优先
- 先剪枝、去重、摘要，再上云。
- 固定字段、固定顺序、固定 JSON。

5. 规则优先执行
- 对高频命令、场景控制、定时自动化优先命中规则，减少 LLM 参与率。

## 6.3 上下文持久化

目标：

- 让系统记住偏好，但不把历史无限堆积。

存储分层：

1. 会话上下文
- 当前场景
- 最近输入
- 最近动作
- 最近澄清结论

2. 结构化偏好
- 温度偏好
- 灯光亮度偏好
- 常用场景时间
- 推荐接受偏好

3. 长期记忆
- “说法 -> 动作”的稳定映射
- 被多次纠正后的偏好结论
- 用户习惯摘要

选型建议：

方案 A：纯 JSON 文件  
优点：简单。  
缺点：并发差、查询弱、后期难维护。

方案 B：`SQLite + JSON 字段 + 少量索引`  
优点：轻量、稳定、适合端侧。  
缺点：需要做基础 DAO 封装。

最终选择：方案 B。  
理由：足够轻量，且明显优于散落 JSON 文件。

写回策略：

- 反馈优先写结构化偏好。
- 长文本历史只保留摘要。
- 低价值日志定期裁剪。

## 6.4 方言与口语化识别

目标：

- 提升家庭场景下的包容性和可用性。

策略：

1. 端侧做高频词归一化
- 热煞了 -> 太热了
- 帮我开哈空调 -> 打开空调
- 灯搞亮点 -> 调亮灯光

2. 维护轻量方言词表
- 按地区逐步扩充。
- 先覆盖温度、灯光、场景、口语祈使四类高频表达。

3. 云端只处理标准化后的语义
- 降低云端误判。
- 保持端侧资源可控。

不建议的路线：

- 当前阶段不自研大规模方言 ASR。
- 不在端侧部署重型语音模型。

## 6.5 TAP 规则的渐进式演进

目标：

- 做最小可用自动化，不把系统整体改成纯 TAP。

适用场景：

- 定时触发
- 环境触发
- 状态触发
- 长期习惯自动化

不适合 TAP 直接接管的场景：

- “有点闷”
- “像昨天晚上那样”
- “我准备睡觉了”
- “我一会儿回家”

最小 TAP 引擎能力：

1. Trigger
- 时间
- 温湿度
- 人员状态
- 场景状态

2. Condition
- 是否有人在家
- 时间段
- 当前设备状态
- 当前场景

3. Action
- 设备控制
- 场景切换
- 通知提示

4. Registry
- 规则注册
- 启停
- 优先级
- 冲突检测

## 7. 从 HausRAGen-main 可迁移的能力

本节是本次补充的重点，记录真实参考点和建议迁移方式。

## 7.1 SVG 上传与户型图元数据

参考实现：

- `D:\HausRAGen-main\server.js`
  - `261-618`：Multer 上传、SVG 校验、尺寸提取、存储逻辑
  - `1312-1318`：上传包装入口
- `D:\HausRAGen-main\data\floor-plans.json`

已验证能力：

- 仅接受 SVG MIME 或 `.svg` 扩展名。
- 上传到 `uploads/floor-plans/`。
- 自动提取 `viewBox / width / height`。
- 元数据写入 `floor-plans.json`。

迁移到 HomeMind 的建议：

- 新增户型图上传接口与元数据存储。
- 保留 SVG 作为唯一受支持格式，先不扩展 PNG/JPG。

在 HomeMind 中的价值：

- 为设备空间可视化提供基础。
- 为规则管理提供“房间/区域”上下文。

## 7.2 设备映射与空间配置

参考实现：

  - `349-396`：`normalizeDeviceMappingToTuples`
  - `664-737`：`GET/POST/DELETE /api/devices`

迁移到 HomeMind 的建议：

- 设备映射格式统一支持：
  - `[["entity_id","area","device_type"], ...]`
  - `{ "devices": [{ "entity_id": "...", "area": "...", "device_type": "light" }] }`
- HomeMind 新增空间配置页。
- 设备映射存储建议拆为：
  - 原始映射
  - 归一化映射
  - 计算后的可视化坐标

在 HomeMind 中的价值：

- 让“设备控制”从抽象列表变成空间化管理。
- 后续规则可直接使用 `area / room` 作为条件。
- 可用于提升设备冲突检查和规则约束。

参考实现：

- `D:\HausRAGen-main\server.js`
  - `1007-1068`：`/api/check-code`
  - `349-396`：设备映射归一化和校验

- 对缩进问题给出警告。
- 对设备映射支持客户端和服务端双重归一化。

迁移到 HomeMind 的建议：

- 命令执行前：
  - 校验 action 类型
  - 校验 target 是否存在
  - 校验 params 范围
- TAP 规则保存前：
  - 校验 trigger / condition / action 完整性
  - 校验涉及设备是否合法
  - 校验规则优先级与冲突
- 设备映射导入前：
  - 客户端先归一化
  - 服务端再强校验

结论：

- 这是必须迁移的能力，不是可选增强。
- 端云协同只要上线，没有 Schema 校验就会很脆弱。

## 10. 验收指标

## 10.1 功能指标

- 明确命令本地直达成功率高。
- 模糊命令端云协同可稳定执行。
- 命令输出通过 Schema 校验后才执行。
- TAP 规则可启用、禁用、删除。
- SVG 户型图可上传并读取尺寸。
- 设备映射可导入、归一化、展示。

## 10.2 体验指标

- 用户可在前端纠正系统结果。
- 用户可在前端管理规则。
- 用户可在前端管理户型图和设备映射。
- 用户能感知系统保留了哪些偏好和记忆。

## 10.3 资源与成本指标

- 业务算法内存占用目标 `1G-2G`。
- 上限不超过 `4G`。
- 云调用仅发生在歧义或模糊意图场景。
- 上传数据为最小必要上下文。
- Token 使用量可统计、可追踪、可优化。

## 11. 风险与应对

### 风险 1：云端调用延迟过高

应对：

- Router 控制上云比例。
- 高频场景本地规则直达。
- 使用本地澄清和本地 fallback。

### 风险 2：云端输出结构不合法

应对：

- 强制固定 JSON。
- 端侧 Schema 校验。
- 不通过则拒绝执行。

### 风险 3：规则、AI、空间配置三者冲突

应对：

- 明确优先级。
- 记录规则命中日志。
- 增加冲突检测和规则禁用能力。

### 风险 4：户型图和设备映射数据越来越乱

应对：

- 统一映射格式。
- 客户端初校验，服务端强校验。
- 原始映射与计算结果分开存储。

### 风险 5：上下文越存越多，影响性能

应对：

- 结构化优先。
- 长文本摘要化。
- 低价值历史定期裁剪。

## 12. 结论

HomeMind 的升级重点应当明确为：

- 前端从“能展示”升级为“能配置、能纠正、能管理”。
- AI 从“本地原型决策”升级为“端云协同、固定 JSON、可校验执行”。
- 记忆从“零散记录”升级为“结构化偏好 + 长期摘要记忆”。
- 自动化从“纯自然语言触发”升级为“自然语言主链路 + 最小 TAP 引擎并行”。
- 空间能力从“无空间语义”升级为“SVG 户型图 + 设备映射 + 房间级自动化基础”。

`D:\HausRAGen-main` 提供的 SVG 上传、设备映射、SSE、压缩、Schema 校验等能力，适合被 HomeMind 吸收为“空间配置、长任务流式输出、上下文压缩、安全校验”的参考实现，但不应整体照搬其产品结构。

最终目标不是把 HomeMind 改成另一个系统，而是在保持当前主链路可用的前提下，把它升级成一个真正可答辩、可演示、可继续开发的家庭智能体产品。
