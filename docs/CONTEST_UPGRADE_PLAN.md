## 当前完成情况（2026-04-23）

本计划中的事项并非全部停留在规划层，当前已形成如下真实状态：

- 已完成：家庭场景主链路、mock 数据联动、基础设备控制与场景切换、最小 TAP 规则 CRUD、SVG 户型图上传、设备映射导入与编辑、基础 Schema 校验、SSE 辅助接口、启动脚本同步、文档迁入 `docs/`。
- 部分完成：Web 原型适配、空间配置直出、户型图渲染链路、上下文压缩、规则自动化执行闭环、前端布局压缩与操作区重构。
- 未完成：端云协同主链路、结构化记忆与偏好持久化、规则冲突检测与命中统计、性能 benchmark、方言归一化、跨场景扩展证明。

后续执行应以前述“未完成”项为优先级更高的落地点，而不是继续扩写概念层内容。

## 1. 计划定位

本计划不再按“理想全量产品”推进，而是基于当前真实代码状态与赛题要求，重新定义为：

- 先把 HomeMind 明确收敛为**家庭场景化智能体系统**
- 先补齐**能证明参赛能力成立**的关键缺口
- 再补增强项，而不是继续堆概念

当前最重要的不是再写更多功能点，而是解决下面三个问题：

1. 哪些赛题能力已经有代码支撑，哪些还没有
2. 哪些内容可以进入答辩材料，哪些当前不能宣称
3. 如何在有限时间内，把“部分符合”提升为“可证明符合”

---

## 2. 当前基线判断

结合现有代码与差距分析，当前 HomeMind 的定位应当是：

- **已形成家庭场景智能体原型**
- **尚未形成完整的跨场景通用智能体系统**
- **尚未完成端侧约束下的量化验证**

### 2.1 已具备能力

- 家庭场景自然语言理解主链路
- 设备控制、信息查询、场景切换等工具调用
- WebSocket 多轮交互与基础状态保持
- 本地知识库与检索增强
- 最小 TAP 引擎基础版
- SVG 户型图与设备映射基础能力
- 规则式上下文压缩基础接口

### 2.2 当前主要缺口

- 缺少“家庭/校园/办公”中的后两类场景能力
- 缺少端侧资源约束的量化证明
- 缺少正式低延迟/高并发验证
- 缺少端云协同主链路
- 缺少结构化上下文与偏好持久化体系
- 缺少方言/口语归一化层
- 缺少可展示的反馈与记忆面板

---

## 3. 新版目标

新版计划只追求三个层级目标。

### 3.1 目标一：把“家庭场景原型”做成“可答辩系统”

达成标准：

- 能清楚说明系统边界
- 能演示核心链路
- 能证明系统不是静态页面和伪逻辑
- 能解释哪些功能已完成、哪些未完成

### 3.2 目标二：把“部分符合”提升到“关键条目符合”

优先补齐以下赛题条目：

- 场景理解与任务规划
- 工具调用与资源整合
- 多轮对话与状态保持
- 有限算力下的实时响应
- 本地知识库与检索增强

### 3.3 目标三：把“无法证明”变成“可量化证明”

优先建立以下可验证证据：

- 端侧内存占用
- 冷启动时间
- 单轮交互延迟
- 并发响应能力
- 规则执行开销
- 知识库检索开销

## 4. 范围收敛

### 4.1 本阶段明确纳入范围

- 家庭场景
- 文字输入
- 浏览器语音输入
- 家居设备控制
- 场景切换
- 规则自动化
- 本地知识库
- 户型图与空间映射
- 端侧性能验证

### 4.2 本阶段明确不纳入范围

- 校园场景正式实现
- 办公场景正式实现
- 摄像头视觉理解
- 复杂图像多模态决策
- 模型蒸馏训练流水线
- 模型量化部署流水线
- 大规模持续学习训练系统

这些内容可以写成“后续扩展方向”，但本阶段不作为必须交付项。

## 5. 技术路线确认

本计划沿用你已经确认的技术决策，不再摇摆。

### 5.1 前端

采用方案 A：

- 继续沿用 `web/client/index.html` 单页增强

理由：

- 当前最快
- 改动最小
- 足够支撑比赛演示和规则页/空间页接入

### 5.2 对话链路

采用方案 A：

- 主链路继续使用 `Socket.IO / WebSocket`

理由：

- 现有代码已接入
- 双向交互天然适合当前聊天与控制

### 5.3 Schema 校验

采用方案 B：

- 结构化 Schema 校验

当前状态：

- 已部分落地

剩余任务：

- 与云端固定 JSON 链路联动
- 补齐更多执行前强校验

### 5.4 空间建模

采用方案 A：

- SVG 户型图 + 设备映射 JSON

当前状态：

- 已基础落地

### 5.5 Token 压缩

采用方案 A：

- 规则式压缩

当前状态：

- 接口已实现

剩余任务：

- 真正接入端云协同请求前处理

### 5.6 TAP 演进

采用方案 B：

- 最小 TAP 引擎作为并行补充层

当前状态：

- 已有基础引擎与规则 CRUD

剩余任务：

- 冲突检测
- 统计信息
- 与空间/房间规则联动

## 6. 新版实施阶段

## 6.1 P0：赛题可证明化

目标：

- 把系统从“能跑”提升到“能证明”

任务：

1. 建立性能测量脚本
- 记录冷启动耗时
- 记录单轮查询耗时
- 记录 `/api/query`、规则评估、知识库检索耗时
- 输出基准报告

2. 建立资源占用测量
- 记录常驻内存
- 记录峰值内存
- 记录 embedding 初始化开销
- 记录知识库加载开销

3. 完成家庭场景边界说明
- 明确系统当前只承诺家庭场景
- 校园/办公写入后续扩展而非当前能力

4. 处理启动与初始化稳定性
- embedding 首次加载卡顿问题
- 首次启动提示与降级机制
- 本地无模型时快速 fallback

交付结果：

- 一份可用于答辩的性能与资源报告
- 一份“家庭场景能力边界”说明
- 一个可稳定启动的演示版本

## 6.2 P1：补齐关键产品能力

目标：

- 把赛题核心能力补齐到“可演示、可解释、可管理”

任务：

1. 端云协同主链路
- 增加 Router
- 定义何时本地执行、何时上云
- 端侧先做最小必要上下文压缩与脱敏
- 云端返回固定 JSON
- 端侧做 Schema 强校验后执行

2. 上下文与偏好持久化
- 会话上下文结构化存储
- 偏好结构化存储
- 长期记忆摘要化存储
- 与知识库写回分层，不再混在一起

3. 前端反馈与记忆面板
- 用户能提交接受/拒绝/更正反馈
- 用户能查看当前保存的偏好摘要
- 用户能删除或重置部分记忆

4. 前端规则管理页补全
- 编辑已有规则
- 展示命中次数
- 展示最近执行时间
- 展示启停状态

5. TAP 最小规则引擎增强
- 规则优先级
- 冲突检测
- 房间级规则支持
- 命中统计

交付结果：

- 端云协同最小闭环
- 结构化上下文与偏好持久化
- 可演示的记忆与规则管理界面

## 6.3 P2：贴近真实家庭环境

目标：

- 提升真实使用场景下的容错与人文体验

任务：

1. 方言与口语归一化
- 建立轻量方言词表
- 常见口语表达归一化
- 模糊意图短句归一化

2. 规则与空间联动
- 让房间/区域成为规则条件
- 支持房间级设备控制建议

3. 长任务补充输出
- 保留 SSE 作为 TAP 生成、导出、长文本任务的补充接口

交付结果：

- 更贴近真实家庭表达习惯
- 更自然的空间化自动化

## 8. 验收标准

新版计划的验收不再只看“功能有无”，而看“是否可证明”。

### 8.1 功能验收

- 用户可通过自然语言控制家庭设备
- 用户可多轮交互并保持上下文
- 用户可配置和管理 TAP 规则
- 用户可上传 SVG 并编辑设备映射
- 用户可查看并修正系统偏好或记忆

### 8.2 技术验收

- 端侧主进程可稳定启动
- 无 embedding 模型时可快速降级
- 规则执行前经过结构化校验
- 云端返回结果必须通过 Schema 校验后执行

### 8.3 证明性验收

- 有基准数据支撑内存与延迟
- 有明确边界说明支撑“当前只做家庭场景”
- 有演示脚本支撑完整业务链路

## 9. 风险与应对

### 风险 1：继续追求“全场景通用”导致项目失焦

应对：

- 当前阶段只答辩家庭场景
- 校园/办公列入后续扩展

### 风险 2：端云协同上线后执行安全性不足

应对：

- 固定 JSON
- Schema 校验
- 设备白名单
- 参数范围校验

### 风险 3：embedding 或知识库初始化拖慢启动

应对：

- 本地优先
- 超时后降级
- fallback 到无向量模式

### 风险 4：记忆体系越做越乱

应对：

- 会话上下文
- 结构化偏好
- 长期记忆摘要

三层分离，禁止继续混写。

### 风险 5：规则、空间、AI 三层相互冲突

应对：

- 增加规则优先级
- 增加冲突检测
- 增加房间级条件验证

## 10. 最终目标表述

本计划完成后，HomeMind 的外部表述应调整为：

- HomeMind 是一个**面向家庭场景的场景化智能体系统**
- 系统具备自然语言理解、工具调用、多轮上下文保持、本地知识增强、规则自动化与空间配置能力
- 系统在端侧资源受限前提下完成了基础性能验证
- 校园、办公、多模态视觉与模型压缩部署属于后续演进方向，而非当前已完成能力

## 11. 一句话总结

新版计划的核心不是“继续加功能”，而是：

- **收敛边界**
- **补齐证据**
- **做实家庭场景**
- **把原型变成可答辩、可证明、可演示的系统**
