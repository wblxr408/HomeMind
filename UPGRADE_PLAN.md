# HomeMind 升级计划书

更新日期：2026-04-23

## 1. 项目背景

HomeMind 当前已经具备以下基础能力：

- 基于 BSR / LSR / LLM 决策的自然语言理解主链路
- 基于 `DeviceController` / `SceneSwitcher` 的统一执行层
- 基于本地知识库的偏好记录与记忆雏形
- 基于 Web 面板的可视化交互入口
- 基于 DQN 的轻量主动推荐骨架

当前问题也较明确：

- 主能力仍偏原型，离“稳定可用的智能家居 Agent”还有距离
- 本地 `mock` 路径可运行，但真实端云协同能力尚未落地
- 上下文记忆、隐私保护、Token 成本控制、自动化规则演进仍不完整
- 前端虽已有壳子，但缺少“可解释、可反馈、可配置”的产品化支撑

因此，本计划书目标是围绕“端云协同 + 上下文持久化 + TAP 渐进式演进 + 体验优化”对 HomeMind 进行系统升级。

## 2. 总体升级目标

本次升级聚焦以下五个方向：

1. 前端 UI 优化
2. 端云协同 AI 部署，完成模糊意图理解
3. 上下文持久化
4. 方言与口语化表达适配
5. TAP 规则的渐进式演进

升级后的 HomeMind 应达到以下状态：

- 在明确命令场景下可本地快速响应
- 在模糊意图场景下可通过端云协同完成理解与执行
- 在断网场景下具备本地兜底能力
- 在长期使用中能够保留上下文、偏好和记忆
- 在自动化场景中引入规则能力，但不破坏原有自然语言主链路
- 在用户体验层面体现“可解释、可配置、有人文关怀”

## 3. 升级原则

### 3.1 保留现有有效架构

不推翻当前主链路，而是在现有架构上增量升级：

- 自然语言主链路仍保留
- 执行层仍保留统一命令结构
- 场景执行仍保留模板化实现
- 轻量学习模块保留，后续再优化

### 3.2 采用端云协同，而不是纯端侧或纯云侧

升级后的职责边界如下：

- 端侧负责：
  - 候选召回
  - 轻量精排
  - 隐私脱敏
  - 上下文管理
  - 本地规则兜底
  - 设备执行
  - 自动化规则执行
- 云侧负责：
  - 模糊意图裁决
  - 参数补全
  - 复杂语义理解
  - 澄清问句生成

### 3.3 云端只给建议，端侧掌握最终执行权

云端模型不直接控制设备。所有云端结果都必须经过端侧：

- schema 校验
- 设备白名单校验
- 参数范围校验
- 高风险动作确认

### 3.4 TAP 规则增量引入，不全量替代当前 AI 主链路

自然语言理解链路仍负责：

- 用户显式输入
- 模糊意图理解
- 复杂语义裁决

TAP 规则只负责：

- 时间触发
- 环境触发
- 状态触发
- 长期自动化场景

## 4. 升级后的总体架构

### 4.1 升级后数据流

```text
用户输入 / 语音输入
-> 端侧预处理
-> 规则直达判断
-> BSR 候选召回
-> LSR 精排
-> 路由器判断是否上云
-> 云端 LLM 返回固定 JSON
-> 端侧 schema 校验
-> DeviceController / SceneSwitcher 执行
-> 记忆写回与状态更新
```

### 4.2 核心模块分层

1. 交互层
- Web UI
- 语音输入
- 用户反馈入口

2. 理解层
- BSR
- LSR
- Inference Router
- Cloud LLM Decider

3. 安全与隐私层
- 隐私脱敏器
- schema 校验器
- 参数范围校验器

4. 记忆层
- 会话上下文
- 结构化偏好存储
- 长期记忆 / 向量补充记忆

5. 自动化层
- TAP 规则引擎
- DQN / 主动推荐

6. 执行层
- DeviceController
- SceneSwitcher
- 协议网关

## 5. 分项升级方案

## 5.1 前端 UI 优化

### 目标

前端升级目标不是“更好看”，而是让系统具备以下产品能力：

- 可解释：用户能看见系统如何理解与执行
- 可反馈：用户能直接修正系统
- 可配置：用户能管理个人偏好和自动化规则

### 建设内容

#### 1. AI 决策流水线面板优化

在现有基础上增强展示：

- 用户输入
- BSR 候选列表
- LSR 排序结果
- 云端 / 本地裁决结果
- 最终执行命令

#### 2. 用户反馈面板

新增标准反馈按钮：

- 接受
- 拒绝
- 更正
- 下次别这样做

作用：

- 驱动本地记忆更新
- 驱动主动推荐学习
- 形成闭环数据

#### 3. 上下文与偏好面板

展示：

- 当前场景
- 最近一次动作
- 最近记忆摘要
- 用户偏好概览
- 近期建议记录

#### 4. 自动化规则管理页

支持：

- 查看现有规则
- 启用 / 禁用规则
- 编辑规则
- 删除规则
- 手动创建简单规则

#### 5. 隐私与云调用说明

前端应明确展示：

- 当前请求是否调用云端模型
- 上传的是哪类摘要数据
- 是否命中本地兜底

### 预期价值

- 增强用户信任感
- 提升系统可解释性
- 强化“智能体正在学习”的产品感

## 5.2 端云协同 AI 部署

### 目标

构建 HomeMind 的核心能力升级：

- 明确命令本地执行
- 模糊命令端云协同理解
- Token 成本可控
- 隐私边界清晰
- 云端结果可控可校验

### 设计原则

#### 1. 端侧先做候选压缩

端侧先通过 BSR / LSR 将动作压缩成 `Top-3` 候选，再决定是否上云。

#### 2. 云端只做有限裁决

云端不做自由规划，不直接调用工具，仅返回固定 JSON：

```json
{
  "action": "设备控制",
  "device": "空调",
  "device_action": "on",
  "params": {"temperature": 26},
  "confidence": 0.91
}
```

#### 3. 端侧必须具备 fallback

若网络异常、云调用失败、云返回不合法，则退回：

- 本地规则执行
- 本地澄清问句
- 本地候选建议

### 路由策略

建议新增 `Inference Router`，路由规则如下：

- 明确命令：本地直达
- `LSR Top1 >= 0.85`：本地直达
- `0.55 <= score < 0.85`：调用云端裁决
- `< 0.55`：直接澄清
- 网络不可用：强制 fallback

### Token 优化策略

#### 1. 剪枝

- 仅上传 `Top-3` 候选
- 上下文只传必要字段

#### 2. 去重

- 候选动作去重
- 上下文摘要去重
- 记忆摘要去重

#### 3. 压缩

只上传：

- 时间段
- 温湿度
- 在家人数
- 当前场景
- 候选动作
- 少量偏好摘要

不上传：

- 完整设备日志
- 家庭成员原始细节
- 长文本历史

#### 4. 输出固定 JSON

云端输出必须强制固定 schema，禁止自由长文本生成。

### 隐私保护方案

#### 1. 数据最小化上传

端侧先抽象成“最小必要上下文”再发云端。

示例：

```json
{
  "hour": 22,
  "temperature": 28,
  "humidity": 72,
  "occupancy": 2,
  "scene": "观影模式"
}
```

#### 2. 不上传敏感原始数据

不上传内容包括：

- 完整家庭设备日志
- 家庭成员详细信息
- 用户原始长期记忆
- 完整日程文本

#### 3. 偏好与历史尽量本地保存

以下内容优先本地存储：

- 用户偏好
- 历史动作记录
- 纠错记录
- 长期记忆

### schema 校验方案

端侧新增命令验证层，校验内容包括：

- 是否为合法字段
- 设备是否在白名单
- 动作是否在白名单
- 参数是否在合理范围
- 场景名是否合法

高风险动作可加入二次确认。

### 预期价值

- 大幅提升模糊意图理解质量
- 避免端侧部署大模型的内存压力
- 满足“允许云侧模型调用”的赛题要求
- 在答辩中同时体现创新性、工程性和成本意识

## 5.3 上下文持久化

### 目标

让 HomeMind 具备“越用越懂你”的长期记忆能力，而不是每次都重新开始。

### 持久化分层设计

#### 1. 会话上下文

保存短期状态：

- 当前场景
- 最近一次用户输入
- 最近一次执行动作
- 最近一次澄清结果

#### 2. 用户偏好

保存结构化长期偏好：

- 常用空调温度
- 睡眠模式灯光亮度
- 常用场景切换时间
- 对主动推荐的接受偏好

#### 3. 长期记忆

保存抽象经验：

- 用户表达与动作之间的映射
- 模糊词与设备偏好的关系
- 被多次纠正后的最终偏好

### 存储策略

#### 结构化优先

优先写结构化偏好表，而不是只写文本。

#### 文本记忆补充

保留少量可检索文本记忆，用于解释模糊语义。

#### 摘要化写回

上下文长期保存时，优先保存摘要而不是原始冗长日志。

### 预期价值

- 提升上下文连续性
- 支撑个性化推荐
- 支撑未来多轮对话演进

## 5.4 方言与口语化表达适配

### 目标

体现家庭场景的人文关怀，增强 HomeMind 对不同说法、非标准普通话和高频方言表达的适应能力。

### 设计边界

本阶段不做大规模方言 ASR 自研，而是做“方言词归一化 + 口语化表达增强”。

### 升级方案

#### 1. 新增方言词归一化层

在 BSR 前增加归一化模块，对高频表达做转换。

示例：

- “热煞了” -> “太热了”
- “帮我开哈空调” -> “打开空调”
- “灯搞亮点” -> “调亮灯光”

#### 2. 建立方言词表

首阶段按重点地区和高频词构建：

- 温度相关表达
- 灯光相关表达
- 场景切换相关表达
- 口语祈使表达

#### 3. 结合云端语义裁决

在标准化后再交给端云协同链路处理，降低误识别率。

### 预期价值

- 增强用户包容性
- 提升家庭场景亲和力
- 形成“技术 + 人文关怀”的差异化亮点

## 5.5 TAP 规则的渐进式演进

### 目标

引入规则自动化能力，但不把整个系统改造成纯 TAP。

### 演进原则

- 保留自然语言主链路
- 新增 TAP 自动化链路
- 最终共用统一执行层

### TAP 适用场景

优先在以下场景中引入规则：

- 定时触发
- 环境触发
- 状态触发
- 长期习惯自动化

示例：

- 晚上 22:30 且有人在家 -> 切换睡眠模式
- 室内温度 > 30 且有人在家 -> 打开空调
- 离家模式启动 10 分钟后 -> 关闭非必要设备

### 不适合 TAP 替代的场景

以下仍由自然语言链路负责：

- “有点闷”
- “像昨天晚上那样”
- “我准备睡觉了”
- “我一会儿回家”

### TAP 引擎最小功能集

#### 1. Trigger

- 时间触发
- 温湿度触发
- 人员状态触发
- 场景状态触发

#### 2. Condition

- 是否有人在家
- 当前时间段
- 当前设备状态
- 当前场景

#### 3. Action

- 设备控制
- 场景切换
- 信息提示

#### 4. Rule Registry

- 规则注册
- 启用 / 禁用
- 优先级
- 冲突检测

### 预期价值

- 提升自动化能力
- 增强智能家居系统完整性
- 避免完全依赖 AI 实时判断

## 6. 技术实施路线

## 6.1 P0：核心可运行升级

目标：先让主价值链路成立。

任务：

1. 接入云端 LLM backend

交付结果：

- 云端结果可被本地校验后执行
- 断网时可 fallback

目标：增强系统可解释性和可配置性。

任务：

1. 前端反馈与记忆面板
2. 前端规则管理页
3. TAP 最小规则引擎
4. 本地记忆与偏好结构化写回

交付结果：

- 用户可看见系统如何理解
- 用户可修正系统
- 用户可管理自动化规则

## 6.3 P2：人文与产品增强

目标：拉开差异化。

任务：

1. 方言词归一化层
2. 方言词表扩展
3. UI 体验进一步打磨
4. 更细粒度的隐私可视化

交付结果：

- 更适合家庭场景
- 更能体现“技术服务于人”的设计理念

## 8. 验收指标

## 8.1 功能指标

- 明确命令本地直达可用
- 模糊命令端云协同可用
- 云端结果能通过 schema 校验
- 上下文可持久化
- TAP 规则可启用/禁用
- 方言高频词归一化可用

## 8.2 体验指标

- 用户能看到 AI 决策过程
- 用户能提交反馈
- 用户能查看和修改规则
- 用户能感知系统记忆在起作用

## 8.3 成本与隐私指标

- 云调用仅发生在模糊意图场景
- 上传数据为最小必要上下文
- 不上传完整家庭原始数据
- Token 使用量可通过日志追踪

## 9. 风险与应对

### 风险 1：云端调用延迟过高

应对：

- 端侧规则兜底
- 本地缓存高频结果
- 仅在歧义场景调用云端

### 风险 2：云端返回非法结构

应对：

- 强制 schema 校验
- 白名单校验
- 参数范围校验
- 不通过则拒绝执行

### 风险 3：上下文持久化越存越乱

应对：

- 结构化优先
- 长文本摘要化
- 明确短期记忆与长期偏好边界

### 风险 4：TAP 规则与 AI 决策冲突

应对：

- 增加优先级
- 增加规则启停
- 记录冲突日志

### 风险 5：方言适配工作量过大

应对：

- 第一阶段只做高频词归一化
- 不做大规模 ASR 自研

## 10. 答辩表述建议

可将本次升级总结为：

> HomeMind 从“原型式智能家居演示系统”升级为“端云协同、可持续学习、可渐进自动化”的家庭智能体。端侧完成候选召回、隐私脱敏、规则兜底和执行控制，云侧完成复杂模糊意图裁决，并通过 schema 校验确保安全可控；同时通过上下文持久化、TAP 演进和方言适配，进一步增强系统在真实家庭场景中的连续性、包容性与实用价值。

## 11. 结论

本次升级不追求盲目堆功能，而是围绕五个关键方向建立 HomeMind 的完整产品形态：

- 在前端上做到可解释、可反馈、可配置
- 在 AI 上做到端云协同、低成本、重隐私
- 在记忆上做到长期持久化
- 在交互上体现人文关怀
- 在自动化上通过 TAP 渐进演进

升级完成后，HomeMind 将更接近一个真正可用、可答辩、可持续扩展的家庭智能家居 Agent，而不是仅停留在架构演示阶段。
